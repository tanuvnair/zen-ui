# zen-ui 7.2.0

*2026-07-18*

Additive. Nothing you call changed and nothing looks different — this release
adds a fourth way to use the same components.

## A fourth binding: `@algorisys/zen-ui-web-components`

zen-ui already shipped as React, Solid and a no-framework vanilla binding. Now
it also ships as **native custom elements**: `<zen-button>`, `<zen-tabs>`,
`<zen-data-table>`, and one for every other component in the library.

```html
<link rel="stylesheet" href="@algorisys/zen-ui-web-components/styles" />
<script type="module">
  import "@algorisys/zen-ui-web-components";
</script>

<zen-button color="primary">Save</zen-button>
<zen-checkbox default-checked></zen-checkbox>
```

They render in the **light DOM**, not behind a shadow root, so the shared
`zen-*` stylesheet and your `--zen-*` token overrides style them byte-for-byte
the same as they style the React and Solid components. There is nothing new to
theme and no second stylesheet to load.

How you drive them:

- **Attributes** for HTML authoring — `<zen-button color="primary" loading>`.
- **JS properties** for anything an attribute cannot carry — objects, arrays and
  callbacks. `dataTable.columns = [...]`, `checkbox.checked = "indeterminate"`.
- **A `json` attribute** for data-driven components when you are writing markup
  by hand rather than script.
- **Handle methods on the element** — `dialog.open()`, `input.focus()`.
- **Events** for value changes — `zen-value-change`, `zen-checked-change` and
  their siblings, dispatched as ordinary `CustomEvent`s.

Because they are just custom elements, they drop into any framework — Vue,
Angular, Svelte, htmx — or a plain HTML page with no build step at all. The same
component set, the same look, wherever you can put an element.

The API mirrors the vanilla binding exactly (the elements wrap those factories),
so the two diverge from React in the same, already-documented ways: data-driven
components such as `Select` and `SegmentedButton` take `items` rather than
compound children, because with no framework there is no context to thread a
parent through.

It is at **alpha**. Try it at the Web Components demo; the components themselves
are the same ones the other bindings have shipped for releases.

## Fixed

- **`bun run dev:all` and `bun run check:size` no longer need `npx`.** Both
  spawned vite through `npx`, which is not present in a bun-only shell — so every
  demo, and the bundle-size gate, failed with a "not found" error that looked
  like something else. They now run vite directly. This only matters if you build
  zen-ui from source; a consumer never saw it.
