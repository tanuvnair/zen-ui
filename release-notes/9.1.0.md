# 9.1.0

You can now control which month a calendar shows. Nothing you already wrote
changes.

## `month`, `defaultMonth`, `onMonthChange` on Calendar

A calendar decides which month to show when it first renders, and then leaves it
alone — so if your app sets `selected` to a date in a **different** month while
the calendar is open, the view stays where the user left it.

That is the right default. Nobody wants the grid jumping out from under them
mid-navigation. But until now there was no way to override it, so if you *did*
want the view to follow, you were stuck.

Three props, matching what the React binding already had:

```jsx
// Open on a specific month, then let the user navigate freely:
<Calendar defaultMonth={new Date(2026, 0)} selected={date} onSelect={setDate} />

// Or drive it yourself — this is how you make the view follow a selection:
<Calendar
  month={month}
  onMonthChange={setMonth}
  selected={date}
  onSelect={(d) => { setDate(d); if (d) setMonth(d); }}
/>
```

`defaultMonth` is read once and still falls back to the month of `selected`,
then to today. `month` makes the view controlled: pass it and the calendar shows
exactly that month, and tells you via `onMonthChange` when the user clicks the
arrows.

## Why this was only missing in two of the four bindings

Worth saying plainly, because it explains an inconsistency you may have hit.

React's `Calendar` is `react-day-picker` with zen-ui styling, and it has always
forwarded that library's full prop set — `month` and `defaultMonth` included. So
React consumers already had this and Solid and vanilla consumers did not.

The two also differ in where they open. react-day-picker computes its first
month as `month || defaultMonth || today` — it does **not** look at `selected`,
so React opens on today's month unless you tell it otherwise. Solid and vanilla
open on the month of `selected`, which is usually what you want.

That difference in the *default* is deliberate and stays. What was wrong was
having no way to override it.
