# zen-ui 8.0.0

*2026-07-19*

Your padded, bordered controls stop overflowing the box they were told to fill.
If you import `@algorisys/zen-ui-*/preflight`, this release changes what your
page measures — including elements zen-ui does not render. That is the whole
release, and it is why the number moved.

## What breaks

**Nothing you call changes. What the browser measures does.**

`/preflight` now sets `box-sizing: border-box` on every element:

```css
*, ::before, ::after { box-sizing: border-box; }
```

Before, an element's declared width was its *content* width and padding and
border were added on top. Now the declared width is the whole thing. Two
consequences:

- **zen-ui's own controls stop overflowing.** `<Input>` is `w-full` plus
  horizontal padding plus a border, so it rendered 26px wider than its
  container. In a flex row that overflow ate the gap between fields, the
  controls ended up touching, and the focus ring — drawn *outside* the element —
  landed on top of the neighbouring field. This was reported as overlapping
  filter inputs. It is fixed.
- **Your own elements are affected too.** The rule is universal, as Tailwind's
  preflight is. Anything of yours that was sized in the content box and relied
  on padding pushing it wider is now that much narrower.

**If you already load Tailwind v3's preflight, nothing changes for you at all** —
you were already getting this rule, and zen-ui was quietly depending on it.

**If you do not, and a layout of yours moves**, the fix is to size the element
the way you now mean it, or opt that subtree back out:

```css
/* only if you have an element that genuinely wants the old measurement */
.my-legacy-widget, .my-legacy-widget * { box-sizing: content-box; }
```

**If you never import `/preflight`**, this release changes nothing you can see —
though it is worth knowing zen-ui's sizing utilities have always assumed
`border-box`, and without that rule from somewhere the components above are
still overflowing.

## Why it is a major bump

The reset is opt-in and has been since 3.0.0, but opting in is not the same as
accepting anything later added to it. A rule on `*` reaches a consumer's own
markup, and pixels that move are a visible change to a component library's
output — the same reasoning that made 7.0.0 major when twelve animations started
running.

## Why the rule was missing

`/preflight` exists precisely because zen-ui's utilities assume Tailwind v3's
element reset is present, and UnoCSS's `presetUno` does not ship one. The file
carried the border reset, the button-chrome reset and the form-font reset from
that same Tailwind preflight — and not the one line the sizing utilities depend
on most. It was the single load-bearing omission, and it looked correct in every
app that also loaded Tailwind.
