# 7.0.0

Your accordions and sheets will start animating. They were supposed to all along.

Two things happened this release, and the first caused the second. zen-ui grew a
third binding — vanilla, no framework, no primitive library — and building it asked
whether the shared core was really framework-agnostic. It was not: twelve
animations had never run, and the answer had been hiding in plain sight since they
were written. The bug is the breaking change; the binding is the new thing.

## What breaks

**Nothing you call changes. Things you look at do.**

Twelve animations start running that never ran before:

- **Accordion** sections now slide open and closed over 200ms. They used to snap.
- **Sheet** panels now slide in and out from whichever edge you opened them on,
  instead of appearing.
- **Dialog**, **Sheet** and their overlays now fade rather than cut.
- **DynamicPage**'s header now collapses smoothly instead of jumping.

No layout moves. No prop changed. Nothing reflows. If your app is fine with
motion, you need do nothing and it will simply look better.

**If you need them off**, the animations are ordinary CSS and respect the platform:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after { animation: none !important; }
}
```

If you had already written that rule, you are unaffected — you were getting the
"no animation" behaviour for the wrong reason, and now you get it for the right one.

This is a major bump because a component library's output is its API, and pixels
that move when they used to be still are a visible change. That is the same reason
3.0.0 was major when the element reset became opt-in.

## Why it never worked

Two independent faults, and either one alone was enough to kill the animation —
which is why neither was noticed.

The animation classes (`zen-anim-accordion-down` and eleven others) were plain CSS
classes, but every component used them behind a state variant
(`data-[state=open]:zen-anim-accordion-down`). UnoCSS can only build a variant of a
utility it owns, and it did not own these, so it generated nothing at all. The
class was spelled correctly, the keyframes were real, and the rule that shipped
matched no element on any page.

Underneath that, the keyframe interpolated height to
`var(--radix-accordion-content-height)` — a Radix-specific name — in the one file
that is shared by every binding. Radix publishes that variable, so React would have
worked once the class did. Kobalte publishes `--kb-accordion-content-height`
instead, so Solid could never have worked, and a binding with no primitive library
publishes neither.

Both are fixed in core. The animations are real utilities now, and the keyframes
read a neutral `--zen-collapsible-content-height` that each binding maps its own
measurement onto.

## Also fixed

- **DataTable, Select, Combobox and every other list**: nothing changed, but the
  Sheet-based surfaces they open now animate as documented.
- **DynamicPage**: the header's collapse transition was written against a class
  UnoCSS does not provide (`zen-transition-[grid-template-rows]`), so it did not
  exist. It does now.

## New: a vanilla binding

`@algorisys/zen-ui-vanilla` is a third binding of the whole library — every
component family React and Solid ship, with no framework and no primitive library
behind it. Props in, a DOM node out:

```js
import { Button } from "@algorisys/zen-ui-vanilla";
import "@algorisys/zen-ui-vanilla/styles";

const save = Button({ variant: "solid", color: "primary", children: "Save" });
document.querySelector("#toolbar").append(save.el);

save.update({ loading: true });   // a targeted DOM write, not a re-render
save.destroy();                   // releases listeners, observers, portals
```

Every factory returns the same handle — `{ el, update, destroy }` — and that is the
whole contract. Where React reaches for compound children and Radix, vanilla is
data-driven: a `Dialog`, `Tabs` or `DataTable` takes its content as data and hands
back one node you own. There is no `DialogContent` to import — there is a `Dialog({
… })` that renders it. Same components, same variants, byte-for-byte the same class
output as the other two bindings.

It is **not on npm yet.** You can see and drive it at the vanilla demo, but this
release publishes no package: the call-site shape is settled, but a few of its
data-driven APIs may still move, and moving them is cheaper than freezing them a
version too early.

Building it is what surfaced the animation bug above, and three more. A binding with
no framework cannot lean on anyone else's primitives, so everything the "shared"
core had quietly borrowed from React showed up the moment it wasn't there.

## Under the hood, if you extend zen-ui

`Button` and `Badge`'s variant tables moved to `@algorisys/zen-ui-core/variants`.
`buttonVariants` and `badgeVariants` are still exported from
`@algorisys/zen-ui-react` and `@algorisys/zen-ui-solid` exactly as before, and the
class strings they produce are byte-for-byte unchanged — verified by diffing the
published stylesheet across the move. You do not need to do anything.
