# 9.5.0

Two new components, in all four bindings. Nothing you already wrote changes.

## Timeline — a sequence of things that happened

An order's history, an audit trail, a ticket's comments.

```jsx
<Timeline
  items={[
    { id: "1", group: "Today", title: "Delivered", timestamp: "09:14",
      state: "success", icon: "check-circle",
      description: "Signed for by R. Iyer at the loading bay." },
    { id: "2", group: "Today", title: "Out for delivery", timestamp: "06:02",
      state: "info" },
    { id: "3", group: "Yesterday", title: "Held at customs", timestamp: "17:40",
      state: "warning" },
  ]}
/>
```

Pass items, not compound children. The shape is always the same — a rail, a
marker, a time, a body — so parts would only let you build one that is subtly
wrong: a marker with no rail, or two rails.

**Grouping is a string you set, not a function that derives it.** You already
know whether two events fall on the same day. Deriving it here would mean
guessing at your timezone and your idea of "today", and getting that wrong is
worse than not offering it.

`density="compact"` **drops** the description and body rather than shrinking the
type, for a sidebar or a popover. In a narrow column a two-line description wraps
to five and the sequence stops being scannable, which was the only reason the
timeline was there.

Four things are decided for you, because they are what makes it readable to
someone who cannot see it:

- It renders an ordered list, so a screen reader announces sequence and length.
- The group heading is not one of the list items, so it does not inflate the
  count that gets announced.
- The rail stops at the last item. A line running past the final event reads as
  "there is more below".
- Markers are hidden from assistive tech — they repeat the title, and "image,
  check circle" before every entry is noise.

`state` is the same five words the rest of the library uses: `default`, `info`,
`success`, `warning`, `error`. Give an item an `icon` and it replaces the dot.
Give it `children` for a body richer than prose — a diff, a quote, a set of
badges.

In the other bindings:

```js
Timeline({ items }).el                          // vanilla
```

```html
<zen-timeline items='[{"id":"1","title":"Delivered"}]'></zen-timeline>
```

## UploadCollection — the list of files, after they are picked

`FileUpload` was the input. Nothing showed the result.

```jsx
<UploadCollection
  items={files}
  onRemove={(item) => remove(item.id)}
  onRetry={(item) => upload(item)}
  onRename={(item, name) => rename(item.id, name)}
/>
```

**It renders the upload; it does not run it.** There is no `url`, no `method`,
no retry policy. You own the transport — `fetch`, XHR, a resumable chunked
client — and write `status` and `progress` back onto the items. `onRetry` hands
the item back to you rather than re-issuing anything itself. A component that
owned the transport would have to guess at your endpoint, your headers, your
auth refresh and your chunk size, and you would spend the rest of the project
fighting it.

An item is `{ id, name }` plus whatever you have: `size`, `status`
(`pending | uploading | complete | error`), `progress`, `error`, `url`,
`uploadedAt`, `uploadedBy`, `thumbnail`. Status defaults to `complete`, so a
document library needs no status at all. A `url` turns the name into a link. A
`thumbnail` replaces the file icon, for a gallery where the name is not what
identifies the file.

**Every button is presence-gated.** No `onRemove`, no delete button. A control
that is always drawn and sometimes does nothing is worse than an absent one,
because the person using it spends a click finding out. In web-components the
gate is the listener:

```js
list.addEventListener("zen-remove", (e) => remove(e.detail.id));
list.addEventListener("zen-rename", (e) => {
  const [item, name] = e.detail;   // already trimmed
  rename(item.id, name);
});
```

Rename edits in place: Enter or clicking away commits, Escape discards. Clicking
away commits deliberately — silently throwing away what someone just typed is the
worse failure.

**A progress bar is drawn only where there is a real number behind it.** An item
that is uploading with no percentage says `Uploading…`, and a queued one says
`Queued`. A bar with no value cannot be read: the underlying primitives disagree
about what one even looks like, one drawing an empty track and the other a full
one, and "full" for a queued file is exactly the wrong answer.

Pair it with `FileUpload` and turn that component's own summary off, so there is
one list rather than two:

```jsx
<FileUpload multiple showFileList={false} onValueChange={setFiles} />
<UploadCollection items={items} onRemove={…} onRename={…} />
```

In vanilla and web-components it is data-driven the same way — push the new array
back with `update({ items })` or `el.items = next`.

## Both

Empty is a state, not an empty box: pass `emptyMessage` and you get a sentence
instead of a bare rail or an empty bordered frame, which read as a component
that failed to load rather than a history with nothing in it yet.
