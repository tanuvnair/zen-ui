# 9.3.0

One new component. Nothing you already wrote changes.

## TreeTable — a table whose rows nest

A cost-centre rollup, an org chart, a bill of materials, a folder tree with
sizes and dates: data where the hierarchy is the point and a flat list would
lose it.

```jsx
import { TreeTable } from "@algorisys/zen-ui-react";

const data = [
  {
    id: "eng", name: "Engineering", headcount: 128, budget: 19_400_000,
    children: [
      { id: "eng-plat", name: "Platform", headcount: 54, budget: 8_200_000,
        children: [ /* … */ ] },
      { id: "eng-qa", name: "Quality", headcount: 13, budget: 2_100_000 },
    ],
  },
];

<TreeTable data={data} columns={columns} />
```

Children come from `row.children` by default — point `getSubRows` elsewhere if
your shape differs. A row with no children is a leaf and gets no chevron.

### The chevron is inside the first column, not in a gutter

This is the part that makes a tree table readable rather than a table with
some toggles bolted to the left. The chevron indents with depth inside the
first column, so the hierarchy reads **down one column**. A gutter of chevrons
at a fixed offset tells you a row expands but not what it belongs to.

### Why it is not a `DataTable` prop

`DataTable` already expands rows, so this looks like it should have been a
flag. It could not be. Hierarchy and grouping want the same three things from
the underlying table — the sub-rows, the expansion state, and the chevron
column — and `DataTable`'s grouping builds its group rows *as* sub-rows. A
table cannot hold a real hierarchy and a grouped one at once; the second would
nest inside the first and mean nothing.

So: reach for `DataTable` when your data is flat and you want grouping,
pagination, column pinning or inline editing. Reach for `TreeTable` when the
data is genuinely nested.

### Searching does not orphan the match

Filtering a tree the obvious way is worse than not filtering it. Search for a
team three levels down and a row-by-row filter drops its division and its
department, leaving a result with no context.

```jsx
<TreeTable data={data} columns={columns} enableGlobalFilter />
```

A surviving row always arrives with its full path. Searching `APAC` gives you
**Go to market → Sales → APAC**, not a lone `APAC`.

### Selecting a parent selects its subtree

```jsx
<TreeTable
  data={data}
  columns={columns}
  getRowId={(row) => row.id}
  enableRowSelection
  rowSelection={selection}
  onRowSelectionChange={setSelection}
/>
```

Untick one descendant and the parent goes **indeterminate** rather than
staying checked — a parent checkbox that contradicts the row below it is worse
than no checkbox. Pass `enableSubRowSelection={false}` when parents are
headings rather than things you can act on.

One thing worth knowing: a parent stays in the selected ids after you untick a
child, and shows indeterminate. So `Object.keys(selection)` can include a
parent whose subtree is only partly selected. If you need "fully selected
only", filter by checking the children yourself.

### Sorting reorders siblings, never levels

Sorting by budget sorts each parent's children among themselves. It does not
flatten the tree into one ranked list — a child outranking its own parent
would have to render somewhere it does not belong.

### Large trees

Only visible rows are ever in the DOM, so a large tree sitting collapsed costs
nothing. If you expand all of a big one, turn on windowing:

```jsx
<TreeTable
  data={bigTree}
  columns={columns}
  enableVirtualization
  maxBodyHeight={360}
/>
```

Measured at 1,240 open rows: 16–25 rows in the DOM instead of 1,240. It needs
`maxBodyHeight` — without a bounded scroller there is no window — and warns in
the console rather than quietly doing nothing if you leave it out.

Not included, deliberately: pagination (a page boundary through a subtree
orphans it) and lazy-loaded children (needs a per-row loading state we would
rather design than guess).

### It is a treegrid

`role="treegrid"`, and every row carries `aria-level`, `aria-expanded` and its
position among its **siblings** — not its position on the page, which would
tell a screen-reader user nothing about the shape. Windowed tables also carry
`aria-rowcount` and `aria-rowindex`, since the DOM no longer holds every row.

Focus roves across rows with a single tab stop: Up/Down move, the forward
arrow opens a closed node then descends into it, the back arrow closes an open
one then climbs to the parent, Home/End jump to the ends. The arrows follow
the writing direction, so in RTL the roles of Left and Right swap and the
indent flips with them.

### All four bindings

`<TreeTable>` in React and Solid, `TreeTable({ … })` in vanilla, and
`<zen-tree-table>` in web-components, where `columns` and `data` are set as JS
properties since they carry render functions and structure.

The vanilla binding has no table library, so its hierarchy engine — the
flatten, the ancestor-preserving filter, the sibling sort and the windowing —
is hand-written. The API and the behaviour are the same; all four were driven
through the same browser checks and produce identical results.
